- design/theme.ts typechecks standalone: `cd design && npx -y -p typescript@latest tsc -p tsconfig.json` (no global tsc; node v25/npm 11 available).
- Showcase harness lives at design/showcase/ (Vite + React + TS + Tailwind v4). It is the design-system runtime — NOT a product app. Build components here. To verify font loading, evaluate `document.fonts.ready` then `document.fonts.check('14px Inter')` / `[...document.fonts].map(f=>f.status)` ('loaded' vs 'error').
- Brand fonts are self-hosted under design/fonts/ (inter/, space-grotesk/, licenses, expo-font.config.ts). Web consumes them via design/fonts.css (@font-face), imported after tokens.css in showcase src/index.css. The @font-face families ("Inter", "Space Grotesk") are what the --font-* tokens reference — never hardcode a font family. RN bundling for these is delivered as design/fonts/expo-font.config.ts (consumed in PRD-03).
- Vite gotcha: design/fonts.css and design/fonts/ live ABOVE the showcase project root, so the dev server 403s their url()s by default. design/showcase/vite.config.ts sets `server.fs.allow: ['..']` (relative to root → allows design/). Avoid `node:url`/`import.meta.url` in vite.config.ts — it's typechecked under tsconfig.node.json which has no @types/node; use a root-relative path string instead.
- Components live at design/showcase/src/components/ (Button, Chip/ChipGroup, Badge), styled by design/showcase/src/components.css (imported last in src/index.css, after tokens.css). PATTERN: inline styles can't do :hover/:active/:disabled, so component visual state lives in a token-only CSS file using real pseudo-classes; pair each pseudo rule with an `.is-force-*` class (and a `forceState` prop) so the showcase can render hover/pressed/disabled statically side by side. Components reference tokens ONLY — verify with `grep -rnE '#[0-9a-fA-F]{3,8}|font-family.*(inter|space|sans-serif)|font-size:.*px' src/components.css src/components/` → only allowed literals are 1px/2px border & focus-outline widths (border-WIDTH isn't a tokenized value; DESIGN.md tokens cover color/type/spacing/radius/shadow). Opacity multipliers, transforms, transition timing, and @keyframes are also non-token literals and fine.
- When a component spec needs a value the core tokens lack, add the token to DESIGN.md FIRST (table + @theme block + theme.ts shape), then design/tokens.css + design/theme.ts, then consume it. US-004 added --text-label(12px)/--text-ui-lg(16px) (+leadings) and --surface-tint-soft/-strong (rgba(0,0,0,.05/.2) ink washes for chip containers & badge fills). DESIGN.md has an internal conflict (button spec says -0.04em but the Do's say -0.02em for 14–18px) — resolved by using the per-role --tracking-* tokens, not a new tracking token.
- Status/semantic colors (--color-status-done/processing/failed) appear ONLY on the Badge's status dot and ONLY for generation state — never decorative. The dot color is applied inline as `var(--color-status-*)` (a token ref, not a literal), selected via a status→token map in Badge.tsx.
- Imagery/photo components (FeedPhotoTile, BeforeAfterCard) take a real `src` (rendered full-bleed via `<img object-fit:cover>`); the showcase has no licensed photos, so it passes a gradient token name via a `placeholder` prop instead — a div with `backgroundImage: var(<token>)`. This is sanctioned by DESIGN.md Imagery (register 2: decorative gradients inside template/preview tiles), NOT the "no gradients on interactive components" rule (which forbids gradient *button fills as brand accents*). Pass a `--gradient-*` token name, never a literal.
- Overlays on photography use the Badge `onPhoto` variant (white text on a --surface-tint-strong frosted pill) so labels read over the image; pair them with the --gradient-scrim overlay (top-down dark fade, the concrete form of the Overlay Ink surface) behind top-corner chips/badges. Both were added in US-005. Reuse onPhoto Badge + scrim for any chip/label sitting on a photo.
- Token-only loading shimmer pattern: base element = --surface-tint-soft; an absolutely-positioned ::before/child band `linear-gradient(90deg, transparent, var(--surface-tint-strong), transparent)` with `transform: translateX(-100%→100%)` via @keyframes. No color literals — the sweep is built from tint tokens (like the US-004 spinner, the keyframes/transform are non-token animation literals and fine).
- Selection state in the black & white system = a 2px `var(--color-midnight-ink)` outline ring (outline, not border, to avoid reflow). Pressed = `transform: scale(0.98)` on :active + an `.is-force-pressed` showcase class. Same forceState pattern as US-004.
- Surfaces with overlay behavior (Sheet/modal) expose an `inline` prop: `inline` renders only the panel (for the static showcase grid), while overlay mode renders a fixed `--surface-tint-strong` scrim + `role="dialog"` panel dismissed by scrim-click / Escape / close button. Reuse this dual-mode pattern for the US-007 paywall (it IS a Sheet/modal surface — build it on top of Sheet). Modal backdrop scrim = `--surface-tint-strong` (rgba(0,0,0,0.2)); never the solid `--surface-overlay-ink`.
- A component prop named `title` collides with React's built-in HTML `title` attribute (typed `string`) when the interface extends `HTMLAttributes`. If `title` must accept `ReactNode`, `Omit<HTMLAttributes<…>, 'title'>` on the interface. Same applies to other DOM-reserved names.
- "Max two type sizes per card" (DESIGN.md) is enforced structurally: Card + Sheet bodies use exactly `--text-heading-sm` (title) + `--text-body` (body). Footer Buttons carry their own control-text tokens and don't count toward the card's content type hierarchy.
- The Sheet body renders in a `<div class="wm-sheet__body">` (not `<p>`) so surfaces built on Sheet (e.g. the US-007 Paywall) can pass rich block content — divs/buttons — legally. Text bodies render identically. Build overlay surfaces (paywall, dialogs, future sheets) ON TOP of Sheet; pass the rich content as `children` and the footer as `actions`.
- "Display type" for prominent numerals (e.g. the paywall price) = the display-size SCALE (`--text-heading-lg` / `--text-display`) in **Inter**, NOT Space Grotesk. DESIGN.md's font table routes "oversized display numerals" to Inter and reserves Space Grotesk for hero headlines/largest section titles only. So a prominent price uses `--text-heading-lg` + `--font-sans` + bold — never the display family. Don't read "display type" as "the display typeface".
- Selectable cards (paywall plans) show selection without reflow by swapping `border-color` (silver hairline → `--color-midnight-ink`) on a fixed 1px border, plus a "black fill accent": a radio that fills `--color-midnight-ink` with a white check (`opacity 0→1`). This is the card-level analog of the US-005 outline-ring selection; use border-color swap when the spec literally calls for a "black border", outline-ring when it must not affect the box.
- The reference showcase page (design/showcase/src/App.tsx) is the single page that assembles the whole system: a token-only `Nav` (in-page table of contents, grouped Foundations / Components) over `GroupHeading` dividers, then Foundations sections (Fonts, Palette, Status colors, Gradients, Type scale, Spacing, Radius, Shadows) then Components sections (Core controls, Feed & before/after, Cards & surfaces, Paywall). Each `Section` takes an `id` for anchor navigation. When you add a new component/section, give its `<Section>` an `id`, add the matching `{ id, label }` entry to the `TOC` array, and keep it under the correct group — that single page is what US-008 verifies and what reviewers use to catch drift.
